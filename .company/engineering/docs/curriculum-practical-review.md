## 開発部門レビュー — 条件付き承認

**強み**: ②追補3規約（識別子アンカー・raw存在禁止・①はゲートに数えない）は 2026-06-09 偽陽性事故の学びの正しい一般化。「不存在チェックはバイパス耐性が本質的に高い」は型哲学とも整合（書かれていないものは偽装できない）。dead check 回避の仕様明文化は 027 ② no-op の再演を構造的に防ぐ。判定エンジン変更ゼロで 48 本に到達する設計判断は健全。

| # | 深刻度 | 指摘 | 対応 |
| --- | --- | --- | --- |
| E-1 | **P1** | **引用符スタイルの偽陰性（040）**: 学習者が `'pending'` とシングルクォートで書くと Union 形②が偽陰性。structure はデリミタを保持するので `["']` クラスで両対応できる。リテラル形を見る②の共通規約に昇格すべき（バッククォート＝テンプレートリテラル型による記述は仕様外とし、hint がダブルクォート形を提示） | ✅ 正本反映（3章規約＋040） |
| E-2 | **P1** | **047 の宣言形が固定されていない**: `let data: unknown` → try 内で代入、という分離スタイルの正答が `: unknown = JSON.parse` アンカーで偽陰性になる。challenge/hint で `const data: unknown = JSON.parse(json)`（try 内で宣言と代入を同時に）の形を明示指定する（015 の T 名指定＝「仕様で書き方を固定」の前例） | ✅ 正本反映（047） |
| E-3 | **P1** | **optional フィールドの不正状態はデータ検証で塞ぐ**: `scenario?`/`requirements?` は「practical なのに scenario 無し」「039 以前なのに requirements 有り」を型で許す。カタログは閉じた静的データなので、verify-lessons に形式整合チェック（040+ は両フィールド必須・039 以前は禁止）を追加して型レベル保証の代替とする。同時に scenario.ts 静的マップのドリフト（040+ 全登録・catalog 外 ID なし・step/total 整合）を vitest で固定 | ✅ 正本反映（6章） |
| E-4 | P2 | **044 never カウントの精密化**: `:[ \t]*never\b` は `never[]` にも一致（`\b` は `[` の前で成立）し、ダミー `const _: never[] = []` ×3 で②カウントを満たせる。代入形 `:[ \t]*never[ \t]*=` にアンカーすれば排除（027 の `const _: never = n` 形）。③で push 実装は強制されるため実害は小さいが②の意図を正確にする | ✅ 正本反映（044） |
| E-5 | P2 | **`as` 不存在チェックの先頭境界統一**: 042/047 の `as[ \t]+...` は理論上 `has Order` 等の部分一致リスク。046 は `\bas` 付き。全レッスン `\bas[ \t]+` に統一 | ✅ 正本反映（042/047） |
| E-6 | P2 | **scenario.ts はシナリオ配列から導出**: lessonId→{title,step,total} の平置きマップは step/total の手書きミスを許す。`{title, lessons: id[]}[]` から step/total を導出する構造にすれば構造的に消える | ✅ 正本反映（2-2） |
| E-7 | P2 | **「こうも書ける」規約の適用**: 判定の都合で書き方を固定する箇所（T 名・default 節維持・JSON.parse 宣言形）は、hint か description で「実務では assertNever ヘルパー等の書き方もある」と必ず補足する（部門ルールの実践クラス適用） | ✅ 正本反映（4章執筆規約） |
